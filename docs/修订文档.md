# Git命名规范

## 1. 提交规范

### 1.1 提交信息格式

提交信息应包含以下部分：

类型(作用域)：描述

例如：

```
feat(apps/user): 添加用户登录功能
fix(apps/user): 修复用户注册时的错误
docs(docs): 更新接口文档
```

### 1.2 类型

- `feat`: 新功能（feature）

    用于提交新功能。
    例如：feat: 增加用户注册功能

- `fix`: 修复 bug

    用于提交 bug 修复。
    例如：fix: 修复登录页面崩溃的问题

- `docs`: 文档变更

    用于提交仅文档相关的修改。
    例如：docs: 更新README文件

- `style`: 代码风格变动（不影响代码逻辑）

    用于提交仅格式化、标点符号、空白等不影响代码运行的变更。
    例如：style: 删除多余的空行

- `refactor`: 代码重构（既不是新增功能也不是修复bug的代码更改）

    用于提交代码重构。
    例如：refactor: 重构用户验证逻辑

- `perf`: 性能优化

    用于提交提升性能的代码修改。
    例如：perf: 优化图片加载速度

- `test`: 添加或修改测试

    用于提交测试相关的内容。
    例如：test: 增加用户模块的单元测试

- `chore`: 杂项（构建过程或辅助工具的变动）

    用于提交构建过程、辅助工具等相关的内容修改。
    例如：chore: 更新依赖库

- `build`: 构建系统或外部依赖项的变更

    用于提交影响构建系统的更改。
    例如：build: 升级webpack到版本5

- `ci`: 持续集成配置的变更

    用于提交CI配置文件和脚本的修改。
    例如：ci: 修改GitHub Actions配置文件

- `revert`: 回滚

    用于提交回滚之前的提交。
    例如：revert: 回滚feat: 增加用户注册功能

### 1.3 作用域

作用域是指提交更改的具体模块或功能区域。为了提高可读性和可维护性，建议在提交信息中明确作用域。

例如：

```
apps/user
pkg/util
docs
```

## 2. 分支命名规范

### 2.1 命名格式

分支命名应遵循以下格式：

```
<类型>/<时间>-<描述>
```

例如：

```
feat/25250708-login
fix/25250708-register-error
docs/25250708-update-docs
```

### 2.2 类型
- `feat`: 新功能分支

    用于开发新功能的分支。
    例如：feat/25250708-login

- `fix`: 修复分支

    用于修复 bug 的分支。
    例如：fix/25250708-register-error

- `docs`: 文档分支

    用于只更新文档的分支。
    例如：docs/25250708-update-docs
